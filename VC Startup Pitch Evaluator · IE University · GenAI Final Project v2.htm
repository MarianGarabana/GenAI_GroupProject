04 Honest Assessment

Powerful — and not infallible.

Every production AI system has failure modes. Here's exactly what can go wrong — and how we built the system to handle uncertainty gracefully.

⚠ What can go wrong

  • !Search results vary.
    DuckDuckGo doesn't guarantee consistent results. On the same deck, different runs returned different articles, different market figures, and different verdicts on the same CEO — leading to composite scores of 2.72 one run and 3.3 the next. Same code, same PDF, same prompt. The web changes; the agent finds different things.
  • !Name ambiguity.
    When searching "Mirza Tahir CEO", the agent returns whoever the internet knows best by that name — which in this case was a religious leader who died in 2003. A founder who shares a name with a more prominent public figure can be unfairly penalised for a contradiction that isn't theirs.
  • !No web presence ≠ fraud.
    Many legitimate early-stage founders aren't indexed on the web. No LinkedIn in search results, no press coverage, no Wikipedia article — not because they're fabricating credentials, but because they're building, not marketing. The system can't tell the difference between "no data" and "false claim."

🛡 How the system handles it

  • ✓Evidence confidence badge.
    Every scoring result shows a High / Medium / Low confidence rating in the Streamlit UI. High means multiple claims were confirmed by specific named sources. Low means evidence was vague, contradictory, or absent. This is surfaced to the analyst explicitly — never buried in the output.
  • ✓Graduated caps, not zeroes.
    The scorer distinguishes between "contradicted" and "unverified." A claim actively contradicted by web evidence scores 0–2. A claim that simply couldn't be found is capped at 5.0 — not zero. A claim backed by strong, named sources scores 7+. The system communicates degrees of uncertainty, not just pass/fail.
  • ✓Human review gate.
    Any score below 6.0 on any individual dimension — not the composite average — freezes the pipeline. The analyst sees the scores, the AI's one-sentence reasoning per dimension, and the evidence quality rating before they write a single word of feedback. The AI flags uncertainty; the human resolves it.
The recommendation (INVEST / PASS) stays consistent across runs even when the exact number shifts. Direction over false precision.
This is the maturity slide — it shows the professor and audience that we understand the limits of what we built. Three failure modes: non-deterministic search results (we literally saw 2.72 vs 3.3 on the same deck), name ambiguity, and missing web presence being mistaken for fraud. Three mitigations: the confidence badge in the UI, the evidence caps in the scorer (5.0 floor for "no evidence", not zero), and the human review gate that forces an analyst to see the reasoning before the memo is final. Close with: the recommendation is stable even when the score isn't — that's by design.

Why an agent over a fixed pipeline?

The right query depends on what the claim actually says — you can't hardcode it in advance. The agent reads each claim and writes its own search. 45 min of analyst Googling → ~30 seconds.

This is the ReAct loop from Session 7 — Reason, Act, Observe, Repeat. The agent reads a claim, writes its OWN query, runs DuckDuckGo or Wikipedia via the ToolNode, reads the result, loops until it can judge. A fixed pipeline would hardcode one query; the agent adapts per claim. Without it, the scorer would grade the startup on its own claims — circular. The agent is what makes the score credible.

Trigger on any score < 6 — not the composite.

market 9 · team 9 · traction 2 · product 8 → composite 7.55. Trigger on the average and this slips through. But traction 2 is a red flag an analyst must see. One weak dimension overrides a healthy average.

LangGraph's most advanced feature. interrupt() pauses execution mid-run and checkpoints state with MemorySaver. Streamlit detects the pause, shows scores + a feedback box, the analyst submits, and Command(resume=...) restarts from the exact same point. Design subtlety: we trigger on ANY individual score below 6, not the composite — a 2 on traction is a flag even when the average looks fine. The audience just watched this fire live in the demo — reference it: "that pause you saw a moment ago, that was interrupt() in action."
05 Engineering Decision · 03

Pydantic as the safety net.

Every Gemini response is validated against a strict typed schema before it enters the pipeline. The model either returns correct, structured output — or it fails loudly.
🧮

Composite in Python

The LLM never does the math. A Pydantic @computed_field applies the VC weighting:

0.30·market + 0.30·team
+ 0.25·product + 0.15·traction
LLMs hallucinate math. Ours doesn't.
🛡️

Strict validation

Recommendation must be INVEST / PASS / CONDITIONAL — exactly. Scores must be floats 0–10. Risks must be a list of 3–5 items, never a blob.

Invalid output raises — never flows through.
🌡️

Deliberate temps

temp = 0 for scoring — same deck, same scores every run. temp = 0.3 for the memo — slightly more natural prose.

Small detail. Deliberate choice.
This is the rubric's guardrails & hallucination mitigation — structured output that can't silently go wrong.
Three points. One: the final score is computed by Python via a Pydantic computed_field — the model proposes four sub-scores; the 30/30/25/15 weighting (team + market dominate at seed) is ours. Two: strict validation — the recommendation Literal accepts only three strings, scores must be floats in range, risks must be a 3–5 item list; bad output raises instead of leaking downstream. Three: temperature 0 for reproducible scoring, 0.3 for readable memos. Exactly the guardrails the brief asks for.
06 RAG · Sessions 5 & 6

Built, tested — and a deliberate decision.

✓ What we built

  • ✓rag/ingest.py — PyPDF loader + RecursiveCharacterTextSplitter (chunk 1000 / overlap 200)
  • ✓rag/retriever.py — Chroma vector store + GoogleGenerativeAIEmbeddings
  • ✓rag/extractor_chain.py — LCEL retrieval chain: query → docs → Gemini
  • ✓rag/rag_demo.py — standalone end-to-end demo
  • ✓tests/test_rag_pipeline.py — 4 unit tests, all passing

◆ Why it's not wired into the live pipeline

Pitch deck size~6,000 chars · ~1,500 tokens
Gemini 2.5 Flash context1,000,000 tokens

The entire deck fits in a single API call — hundreds of times over. RAG solves documents that are too large for the context window. That problem doesn't exist here, so chunking would only add latency and retrieval risk, with no quality gain. Confirmed with the professor.

A strength, not a gap. We built the full RAG pipeline: PyPDF loader, recursive splitter, Chroma with Gemini embeddings, an LCEL retrieval chain, a demo, passing tests. Then we decided not to wire it in: a deck is ~6,000 characters; Gemini 2.5 Flash has a million-token window. The deck fits in one call hundreds of times over. RAG handles documents too big for context — that problem isn't present, so chunking only adds latency and a chance of retrieving the wrong chunk. Kept as a documented concept demo; confirmed with the professor. Knowing when NOT to use a tool is the point.
